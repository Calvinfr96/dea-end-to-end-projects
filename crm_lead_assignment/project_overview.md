# CRM Lead Assignment Project

## Overview
This project will use AWS services to build a system that automates the capture of newly-created leads using CRM webhooks, waits for a short buffer period (e.g. 10 minutes), and assigns the correct lead owner based on CRM-updated information. The system must also notify sales teams in real time with the enriched lead data via Slack or email.
- In sales-driven organizations, new leads are typically generated by a Customer Relationship Management (CRM) system. These systems often introduce gaps between when a lead is created and when it is assigned to a sales representative. This delay can create inefficiencies and missed opportunities if leads are not followed up promptly.
- This project aims to automate the capture of newly-created leads using CRM webhooks, wait for a short buffer period (e.g. 10 minutes), and assign the correct lead owner based on CRM-updated information.
- The system must also notify sales teams in real time with the enriched lead data via Slack or email.
- Objectives:
    1. Capture real-time lead creation events using CRM webhook functionality.
    1. Store the lead data in S3 in its raw format.
    1. Introduce a 10-minute delay to allow the CRM to assign a lead owner.
    1. Perform a lookup against another S3 bucket that contains updated lead-owner mappings.
    1. Enrich the raw lead data with the assigned lead owner and store the data in S3
    1. Send a notification via Slack or email, including the lead’s name, ID, create date, email, owner, and funnel.

## Data Engineering Considerations
- The data pipeline will be handling small amounts of data in real-time. The data will be stored in a data warehouse, not a data lake, as we will be working with manageable amounts of semi-structured (JSON) data, not large amounts of unstructured data.
    - Since the data is semi-structured, an ETL process will be used to handle the data instead of ELT.
- Business requirements:
    - Design an automated data pipeline that receives and stores raw lead information, merges it with processed lead information after a small delay period, and sends out lead notifications to the sales team for timely follow up.
- To build the data pipeline, we can take advantage of GitHub Actions Workflow, AWS Open ID Connect (OIDC), and AWS Cloud Development Kit (CDK). Doing so allows us to write our infrastructure as code, instead of manually creating it. Doing so vastly improves maintainability.
- Establishing a connection with Close Webhook required creating a subscription URL with Zapier. When this URL receives a lead notification from Close CRM, it sends the data to an S3 bucket.
- From S3, establishing a 10-minute delay between raw lead data arriving and being processed with updated lead data, SQS can be used with S3 event notifications. An event notification can be sent to SQS, which implements a 10-minute delay before send a message to a compute service to process the raw data with the updated data.
- Using Lambda as a compute service makes the most sense because it is serverless and lightweight, making it effective for small workloads and low data volumes. Lambda also interacts natively with SQS and can be quickly spun up and shut down _after_ the 10-minute delay, eliminating idle compute time.
    - For heavier data transforms, Lambda would not be suitable because of its memory and execution time limits. Services such as AWS Glue or Amazon EMR would be a better fit.
    - For continuous streaming scenarios, where thousands of leads are being processed every second, Lambda's pay-per-request model becomes ineffective. Services such as Amazon ECS Fargate would be a better fit because they can be run continuously.
    - For complex, multi-step retries, orchestration logic cannot be cleanly coded into a single Lambda function. AWS Step Functions would need to be used to orchestrate several Lambda functions.
- Using Lambda to perform the data processing and message delivery makes more sense than using SNS separately for message delivery because the messages need to be customized based on the processed data. This can be done in the Lambda function after the data is sent to the S3 bucket.

## System Design Considerations
- Target Database: S3
    - Provides simple and scalable storage of raw and processed data at low cost.
- Data Extraction/Ingestion: Zapier Webhook
    - The webhook receives alerts from Close CRM and sends the raw lead information directly to S3. Other alternatives, such as API Gateway, are available. However, these services do not integrate as tightly with the Close CRM as Zapier.
- Data Transformation/Loading: Lambda
    - A Lambda function can process the raw lead data after receiving an S3 Event Notification routed through SQS, which implements a 10-minute delay. Once the data is processed, it can be sent to a separate S3 bucket.
- Scheduling Mechanism:
    - Data processing is triggered by an S3 Event Notification routed through SQS. SQS implements a 10-minute delay needed for a lead's information to be updated as it is assigned to an owner, before it is assigned to a sales representative.
- CI/CD and Dev Ops: GitHub Actions
    - Seamlessly integrates with AWS and automatically updates infrastructure, ingestion scripts, and transformation scripts whenever a commit is pushed to the source repository.
- Error Handling:
    - If the Lambda code crashes due to network timeouts, unhandled syntax errors, or downstream API disconnects (such as Slack being down), the state engine flags that execution as `Failed`.
    - The Step Functions Console generates an interactive, color-coded node execution log. Red steps clearly indicate exactly which state broke, accompanied by a complete stack trace of the JavaScript exception,
    - Because each file drop instantiates a completely separated, decoupled workflow execution, a single bad file payload will never jam or block other incoming leads from processing simultaneously.
- Retries:
    - The Lambda function is configured with a 15-minute runtime cutoff. This guarantees it will cleanly exit instead of running indefinitely and over-billing your account if a public HTTP request stalls.
    - If an execution fails because an external resource was missing (e.g., your Lambda function looked for a public `.json` file that hadn't finished writing yet), the failed workflow can be re-driven to re-run the step with the exact same payload.
    - The Lambda invocation step function can be automatically retried before failing.
- Logging:
    - Everything written using standard JavaScript outputs (`console.log`, `console.error`, `console.warn`) inside your Node.js code is intercepted and securely routed directly to an automatically managed CloudWatch Log Group named `/aws/lambda/LeadProcessorLambda`.
    - Because the file paths are predictably derived directly from the payload string (`crm_event_{lead_id}.json`), CloudWatch logs can be searched using a specific `lead_id` to quickly locate the entire processing timeline across S3, Step Functions, and Slack.
